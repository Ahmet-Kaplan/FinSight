# FinSight 优化 TODO（v2：实现方案 + 开发计划）


1. 提升扩展性: 现在的报告只能针对三个固定的level来撰写，希望支持更多种类的报告。类似skills的配置，用户只需要给一个文件夹，里面写上希望写的报告类型或者markdown example，就能自动适配。这需要把现在的prompt做成模块化的。或者基于用户的模板来进行自动学习，提升整体的泛化性。
2. 优化agent的实现: 提升并发性和运行效率。现在agent的实现还不太统一，没有一个明确的架构。而且现在多智能体的实现更像是一种线性的workflow，实现上不够优雅，显得比较杂乱。我想统一成类似agent swarm那种。
3. 优化报告的成本，现在的报告撰写过长，考虑增加更精简、密度更高的版本
4. codebase本身的实现上优化，提升稳定性、代码的整洁性
5. 支持skill的注入，用户提前写好想分析的路径/思路，agent系统能够直接follow。 或者把现在的内容/流程整合成skill
6. 交互上引入人机协同

架构方面:
1. 统一抽象: 报告模板规范， 注入规范，任务图规范，agent运行流程规范；
2. 插件化报告+agent swarm运行；引入任务图调度与并发执行；
3. 成本与交互优化。

实现base agent

实现agent base能力: context压缩，tool-use， memory-as-action

目标：用户配置好后agent能够直接使用

阶段0：
构建好整体报告生成的统计链路，记录一份报告生成过程中的各种指标(api调用，token数量、任务数量等)

阶段1：
优化agent架构： 
(1) 拆分出各类agent角色，并且统一各类agent的实现 (让所有agent都基于同一个架构，不要有的用function call，有的通过code executor来调用). 统一通过code executor这种架构来调用工具，方便管理上下文以及所有agent的变量空间。
(2) 优化现在的线性workflow，参考kimi agent的agent swarm架构构建多智能体（你需要上网调研一下）；将报告生成升级为智能调度的任务图模式，增加现在流程的灵活性。
(3) 优化现有的并发执行、重试、超时等机制，增强整体的稳定性和易用性。

阶段2:
(1) 报告插件化: 把agent的prompt做成模块化的，梳理现在的agent的prompt，把跟报告风格、内容相关的prompt拆出。用户可以轻松接入新类型的报告或者新topic的报告，通过新增example报告就能自动学习，泛化出新的prompt。
(2) 智能理解用户需求： 用户不止能调整内容方向、写作风格，还能调整输出的粒度等。全部通过自然语言来实现，而不要引入太多的显式config。 用户提前写好想分析的路径/思路，agent系统能够直接follow，相当于用户的需求可以直接影响运行图的构建。


阶段3:
增强整体事实性: 引入事实核验和冲突检验，并增加溯源机制； 
主要是优化最终的报告撰写智能体以及数据分析智能体；让agent在每一步决策的时候能够带上细粒度引用；

阶段4：
优化UI交互界面，在demo上加入人机协同的入口。
